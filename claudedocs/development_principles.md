# 개발 원칙 및 선호사항

이 문서는 사용자의 개발 철학과 선호사항을 기록합니다.

---

## 핵심 원칙

### 1. YAGNI (You Aren't Gonna Need It) 엄격 적용
- **불필요한 코드는 즉시 제거**
  - 중복 필드, 사용하지 않는 변수, 추상화 모두 제거

- **미래를 위한 추상화 금지**
  - 현재 필요한 것만 구현
  - "나중에 필요할 수도" → 필요 없음

### 2. 타입 안전성 중요
- **적절한 Python 타입 사용**
  - 문자열보다 datetime, enum 등 명확한 타입 선호
  - 직렬화는 Pydantic 등이 알아서 처리
  - 예: `published_at: datetime`, `channel: Channel(Enum)`

- **타입 힌트 필수**
  - 함수 시그니처에 타입 명시
  - Optional 명확히 표시

### 3. 실용성 우선
- **실제 사용 패턴 기반 결정**
  - "insert만 사용" → 복잡한 ORM 불필요
  - 이론보다 실제 필요성 우선

## 코드 스타일

### 간결성
- **불필요한 주석 제거**
  - 자명한 내용은 주석 불필요

- **간단한 코드 선호**
  - 환경 변수보다 기본값 직접 설정 (StateStore 방식)
  - 복잡한 분기보다 명확한 로직

### 테스트
- **Mock 테스트 금지**
  - 실제 통합 테스트 선호
  - 예: 실제 RSS 피드 호출하는 테스트

### 네이밍
- **명확하고 간결한 이름**
  - 중복되는 개념은 하나로 통일
  - 불필요한 prefix/suffix 제거

## 의사결정 방식

### 1. 솔직한 의견 요구
- **"솔직히 말해봐. 객관적으로 어떤게 나은지"**
  - 돌려 말하지 말고 직접적으로 판단
  - 장단점 명확히 제시
  - 추천하는 방식 명확히 표현

### 2. 변경 전 논의
- **복잡한 해결책 사용 전 질문**
  - 예: TYPE_CHECKING 같은 트릭 사용 전
  - 순환 참조 해결 방법 여러 개 제시하고 선택하게

### 3. 단계적 검증
- **큰 변경은 단계별로 확인**
  - 예: Database 연결 먼저 확인하고 통합
  - 한 번에 다 하지 말고 나눠서 검증
